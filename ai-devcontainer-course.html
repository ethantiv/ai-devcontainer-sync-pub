Moduł 01

O co tu chodzi

Otwierasz kontener — wszystko jest gotowe. Claude Code, pluginy, serwery MCP, system autonomicznych pętli. Nie musisz niczego instalować ręcznie.

Twój zestaw narzędzi

Po uruchomieniu środowiska masz dostęp do:

  • Claude Code — agent AI do programowania w terminalu
  • loop — komenda uruchamiająca Claude w autonomicznych cyklach (projektuj → planuj → buduj)
  • Pluginy i skills — gotowe umiejętności: tworzenie commitów, review kodu, praca z PDF/DOCX, diagramy, i więcej
  • Serwery MCP — Claude łączy się z zewnętrznymi serwisami (dokumentacja bibliotek, Coolify, AWS)

Struktura repozytorium

ai-devcontainer-sync/ config/env-config.yaml ← główna konfiguracja config/.env ← Twoje tokeny (nie commitowane) src/ ← system loop (komenda loop) docker/ ← Docker / Raspberry Pi setup-local.sh ← instalacja na macOS
Kluczowa zasada

Zmieniasz jeden plik (config/env-config.yaml) — reszta synchronizuje się automatycznie. Pluginy, skills, serwery MCP, tożsamość Git — wszystko jest tam zdefiniowane.

Moduł 02

Jak uruchomić

Trzy kroki do działającego środowiska — niezależnie od wybranej opcji.

Krok 1: Stwórz plik .env

Skopiuj przykład i wpisz swój token GitHub. To jedyna wymagana zmienna.

cp config/.env.example config/.env
# Edytuj config/.env — ustaw GH_TOKEN

Krok 2: Uruchom środowisko

▶ VS Code / Codespaces (DevContainer)

Otwórz repozytorium w VS Code i kliknij Reopen in Container. Gotowe.

▶ Docker (RPi / standalone) 
docker compose -f docker/docker-compose.yml up -d
docker exec -it claude-code bash
▶ macOS (bez Dockera) 
./setup-local.sh

Krok 3: Sprawdź, czy działa

claude mcp list               # czy serwery MCP są podpięte
claude plugin marketplace list # czy pluginy się zainstalowały
Moduł 03

Komenda loop

loop to główne narzędzie. Uruchamia Claude w autonomicznych cyklach — Ty podajesz pomysł, Claude planuje i buduje.

Inicjalizacja w projekcie

Wejdź do swojego projektu i uruchom loop init. Stworzy katalog loop/ z promptami i konfiguracją.

cd ~/projects/mój-projekt
loop init                     # podstawowa inicjalizacja
loop init --type web          # + skills do React/Next.js
loop init --type devops       # + skills do Terraform
loop init --type fullstack    # web + feature-dev
loop init --list-types        # pokaż dostępne typy

Główne komendy

Co chcesz zrobićKomenda
Przemyśleć rozwiązanie z Claude (interaktywnie)loop design
Stworzyć plan implementacjiloop plan
Zrealizować plan (autonomicznie)loop build
Planowanie + budowanie od razuloop run
Sprawdzić zdrowie instalacjiloop doctor
Statystyki ostatniego uruchomienialoop summary
Zbić procesy na portach dev (3000, 5173...)loop cleanup

Podpowiadanie pomysłów (-I)

Przekazujesz Claude pomysł na feature — tekst, plik, lub URL do GitHub issue:

loop run -I "Dodaj autentykację JWT"
loop plan -I @docs/specyfikacja.md
loop plan -I https://github.com/org/repo/issues/42
Moduł 04

Design → Plan → Build

Trzy fazy autonomicznego programowania. Możesz użyć wszystkich trzech lub przeskoczyć od razu do loop run.

Trzy fazy

Faza 1
Design
Rozmowa z Claude o wymaganiach i architekturze
Faza 2
Plan
Claude tworzy plan z taskami i checkboxami
Faza 3
Build
Claude realizuje plan, testuje i commituje

Kiedy czego używać

SytuacjaCo uruchomić
Prosty feature, wiesz czego chceszloop run -I "opis"
Chcesz najpierw przemyśleć podejścieloop design → loop plan → loop build
Masz spec w plikuloop run -I @docs/spec.md
Feature z GitHub issueloop run -I https://github.com/.../issues/42
Poprzedni plan ukończony, nowy cyklloop plan --new -I "następny feature"

Co robi Build w każdej iteracji

  1. Bierze pierwszą niezakończoną fazę z planu
  2. Pisze testy → implementuje → sprawdza testy
  3. Odhacza checkboxy w planie
  4. Commituje i pushuje
  5. Jak plan jest w 100% zrobiony — kończy
Przydatne flagi

-i 20 — ogranicza liczbę iteracji build
--no-early-exit — nie kończy po zakończeniu planu
--interactive — tryb ręczny (Claude czeka na Twój input)

Moduł 05

Pluginy i skills

Pluginy i skills to umiejętności, które Claude ładuje automatycznie gdy ich potrzebuje. Nie musisz ich wywoływać ręcznie — z wyjątkiem slash commands.

Slash commands (wpisujesz Ty)

Te komendy wpisujesz bezpośrednio w rozmowie z Claude:

KomendaCo robi
/commitTworzy commit z opisem zmian
/commit-push-prCommit + push + otwórz Pull Request
/roadmapGeneruje lub aktualizuje plan projektu
/git-worktree:createNowy branch w odizolowanym katalogu
/dependency-updateAktualizuje zależności projektu
/read-arxiv-paperStreszcza artykuł naukowy z arXiv

Pluginy (Claude używa sam)

Claude automatycznie ładuje odpowiedni plugin gdy tego potrzebuje:

PluginKiedy się włącza
superpowersPlanowanie, TDD, debugging, code review
feature-devRozbudowa featureów z eksploracją kodu
frontend-designTworzenie interfejsów UI
code-simplifierUpraszczanie i refaktoryzacja
hookifyTworzenie reguł automatyzacji

Skills (specjalistyczne umiejętności)

Instalowane z zewnętrznych repozytoriów. Przykłady:

  • pdf, docx, xlsx — praca z plikami biurowymi
  • agent-browser — Claude otwiera i obsługuje przeglądarkę
  • beautiful-mermaid — renderowanie diagramów
  • terraform-* — styl, testy i refaktoryzacja Terraform
  • humanizer — usuwanie stylu AI z tekstu

Serwery MCP (połączenia z serwisami)

Claude łączy się z zewnętrznymi serwisami przez protokół MCP:

SerwerDo czego służy

Moduł 06

Pliki konfiguracyjne

Dwa pliki definiują całe środowisko. Nie musisz tu zaglądać codziennie — ale warto wiedzieć gdzie co zmienić.

config/.env — Twoje sekrety

Tokeny i klucze API. Nigdy nie jest commitowany.

ZmiennaOpis
GH_TOKENwymaganyToken GitHub (repo + workflow)
CONTEXT7_API_KEYopcjaDokumentacja bibliotek przez MCP
COOLIFY_BASE_URLopcjaURL instancji Coolify
COOLIFY_ACCESS_TOKENopcjaToken Coolify
GH_TOKEN_WORKopcjaToken do konta firmowego GitHub
BACKUP_PINopcjaPIN do szyfrowania backupów

config/env-config.yaml — cała reszta

Tu definiujesz: tożsamość Git, listę pluginów, skills, serwery MCP. Podzielony na defaults (wspólne) i environments (per środowisko).

▶ Dodaj plugin

Dopisz nazwę do listy plugins.marketplace:

plugins:
  marketplace:
    - superpowers
    - commit-commands
    - nowy-plugin        # ← dopisz tutaj

Po zmianie: przebuduj kontener lub uruchom setup-env.sh.

▶ Dodaj skill

Dopisz wpis do listy skills:

skills:
  - url: "https://github.com/autor/repo"
    name: nazwa-skill
▶ Dodaj serwer MCP

Dopisz w sekcji environments.<twój-env>.mcp_servers:

mcp_servers:
  - name: mój-serwer
    type: stdio
    command: npx
    args: ["-y", "@org/serwer@latest"]
    env:
      API_KEY: "${MÓJ_KLUCZ}"
Uwaga

Dla zdalnych serwerów używaj type: "http", nie "url" — "url" cicho nie działa.

▶ Konto firmowe GitHub

Jeśli masz oddzielne konto GitHub do pracy (SSO), środowisko automatycznie routuje tokeny:

git:
  work:
    email: "imie@firma.com"
    orgs: "TwojaOrg|InnaOrg"
    dirs:
      - "~/projects/Praca"

+ ustaw GH_TOKEN_WORK w .env.

Moduł 07

Ściągawka komend

Wszystko w jednym miejscu. Kliknij Copy i wklej do terminala.

Pierwsze uruchomienie

cp config/.env.example config/.env    # stwórz plik z tokenami
# edytuj config/.env — ustaw GH_TOKEN
claude mcp list                       # czy serwery MCP działają
claude plugin marketplace list        # czy pluginy się zainstalowały

Nowy projekt z loop

cd ~/projects/mój-projekt
loop init                             # zainicjalizuj loop
loop init --type web                  # z webowymi skills
loop doctor                           # sprawdź instalację

Codzienna praca

loop run -I "Dodaj autoryzację JWT"    # pomysł → plan → build
loop design                           # interaktywny brainstorming
loop plan                             # tylko planowanie
loop build                            # tylko budowanie
loop build -i 20                      # max 20 iteracji
loop build --interactive              # tryb ręczny
loop plan --new -I "Następna rzecz"    # nowy cykl
loop summary                          # statystyki
loop cleanup                          # zbić dev serwery

Po zmianach w konfiguracji

./.devcontainer/setup-env.sh           # re-synchronizuj środowisko
loop update                           # odśwież symlinki loop

Backupy

.devcontainer/backup.sh create         # zaszyfrowany backup
.devcontainer/backup.sh restore plik   # przywróć z backupu
.devcontainer/backup.sh list           # lista backupów
