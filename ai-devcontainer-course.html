Moduł 01

O co tu chodzi

Otwierasz kontener — wszystko jest gotowe. Claude Code, pluginy, serwery MCP, autonomiczny tryb loop. Nie musisz niczego instalować ręcznie.

Twój zestaw narzędzi

Po uruchomieniu środowiska masz dostęp do:

  • Claude Code — agent AI do programowania w terminalu
  • loop — komenda uruchamiająca Claude autonomicznie: design (brainstorming) i run (plan + build + commit w jednej sesji)
  • Pluginy i skills — gotowe umiejętności: commity, review kodu, praca z PDF/DOCX/XLSX, diagramy Mermaid, Terraform, i więcej
  • Serwery MCP — Claude łączy się z zewnętrznymi serwisami: dokumentacja bibliotek (context7), Jira/Confluence (atlassian), AWS (Lambda, IaC, pricing)

Struktura repozytorium

ai-devcontainer-sync/ config/env-config.yaml ← główna konfiguracja config/.env ← Twoje tokeny (nie commitowane) src/ ← system loop (komenda loop) docker/ ← Docker / Raspberry Pi setup-local.sh ← instalacja na macOS
Kluczowa zasada

Zmieniasz jeden plik (config/env-config.yaml) — reszta synchronizuje się automatycznie. Pluginy, skills, serwery MCP, tożsamość Git — wszystko jest tam zdefiniowane.

Moduł 02

Jak uruchomić

Trzy kroki do działającego środowiska — niezależnie od wybranej opcji.

Krok 1: Stwórz plik .env

Skopiuj przykład i wpisz swój token GitHub. To jedyna wymagana zmienna.

cp config/.env.example config/.env
# Edytuj config/.env — ustaw GH_TOKEN

Krok 2: Uruchom środowisko

▶ VS Code / Codespaces (DevContainer)

Otwórz repozytorium w VS Code i kliknij Reopen in Container. Gotowe.

▶ Docker (RPi / standalone) 
docker compose -f docker/docker-compose.yml up -d
docker exec -it claude-code bash
▶ Coolify (deploy na serwerze)

Stwórz zasób Docker Compose w Coolify, ustaw Base Directory: /docker i Docker Compose Location: /docker-compose.yml. Dodaj zmienne środowiskowe w ustawieniach aplikacji.

▶ macOS (bez Dockera) 
./setup-local.sh

Krok 3: Sprawdź, czy działa

claude mcp list               # czy serwery MCP są podpięte
claude plugin marketplace list # czy pluginy się zainstalowały
Moduł 03

Komenda loop

loop to główne narzędzie. Przygotowuje projekt do pracy z Claude i uruchamia go z odpowiednim promptem — Ty podajesz pomysł, Claude planuje i buduje.

Inicjalizacja w projekcie

Wejdź do swojego projektu i uruchom loop init. Stworzy katalog loop/ z symlinkami do skryptów i skopiuje szablony promptów (które możesz edytować).

cd ~/projects/mój-projekt
loop init                     # podstawowa inicjalizacja
loop init --web               # + skills do React/Next.js
loop init --devops            # + skills do Terraform

Główne komendy

Co chcesz zrobićKomenda
Przemyśleć rozwiązanie z Claude (interaktywnie)loop design
Autonomiczne planowanie + budowanie (obie fazy)loop run
Tylko faza planowanialoop run --plan
Tylko faza budowania (używa istniejącego planu)loop run --build
Zbić wszystkie procesy looploop kill
Odświeżyć symlinki po aktualizacjiloop update

Podpowiadanie pomysłów (-i)

Przekazujesz Claude pomysł na feature — tekst, plik, lub URL do GitHub issue:

loop run -i "Dodaj autentykację JWT"
loop run -i @docs/specyfikacja.md
loop run -i https://github.com/org/repo/issues/42
Flagi

-i / --idea — przekazuje pomysł (tekst, @plik, URL)
-n / --new — archiwizuje obecny plan i zaczyna od nowa
--plan — uruchamia tylko fazę planowania
--build — uruchamia tylko fazę budowania (wymaga istniejącego planu)
--interactive — tryb ręczny (Claude czeka na Twój input)
--tmux — uruchom w oddzielonej sesji tmux

Moduł 04

Design → Run

Dwie fazy autonomicznego programowania. Możesz użyć obu lub przeskoczyć od razu do loop run.

Dwie fazy

Opcjonalna
Design
Interaktywna rozmowa: brainstorming i design doc
Główna
Run
Dwie pod-fazy: plan → build → commit. Możesz uruchomić je osobno flagami --plan / --build.
Jak to działa

Każda faza to jedno wywołanie Claude CLI z promptem (PROMPT_design.md lub PROMPT_run.md). Claude czyta prompt, ładuje skills i wykonuje całą pracę w jednej sesji. Nie ma pętli ani iteracji — Claude sam decyduje ile kroków potrzebuje.

Co robi Design

  1. Ładuje skills z PROMPT_skills_design.md (domyślnie: brainstorming)
  2. Czyta docs/IDEA.md oraz istniejące design docs
  3. Prowadzi interaktywną rozmowę o wymaganiach i architekturze
  4. Zapisuje design doc do docs/
  5. Commituje i pushuje
Uwaga

Design jest zawsze interaktywny — Claude rozmawia z Tobą. Nie tworzy planu ani nie implementuje.

Co robi Run

  1. Ładuje skills z PROMPT_skills_run.md (writing-plans, subagent-driven-development, agent-browser, auto-revise-claude-md)
  2. Czyta docs/IDEA.md i design docs jako kontekst
  3. Tworzy plan implementacji (skill writing-plans)
  4. Realizuje plan, używając równoległych subagentów (skill subagent-driven-development)
  5. Aktualizuje CLAUDE.md o nowe wnioski
  6. Commituje i pushuje
Uwaga

Run jest domyślnie autonomiczny — Claude pracuje bez pytania. Użyj --plan, żeby zatrzymać się po planowaniu i przejrzeć plan przed budowaniem, a następnie --build, żeby kontynuować. Użyj --interactive, żeby móc ingerować w proces. Po zakończeniu, loop.sh automatycznie commituje niescommitowane zmiany.

Moduł 06

Pliki konfiguracyjne

Dwa pliki definiują całe środowisko. Nie musisz tu zaglądać codziennie — ale warto wiedzieć gdzie co zmienić.

config/.env — Twoje sekrety

Tokeny i klucze API. Nigdy nie jest commitowany.

ZmiennaOpis
GH_TOKENwymaganyToken GitHub (repo + workflow)
SSH_PRIVATE_KEYopcjaKlucz SSH (base64) do uwierzytelniania Git
GIT_USER_NAMEopcjaImię i nazwisko do commitów
GIT_USER_EMAILopcjaEmail do commitów
CONTEXT7_API_KEYopcjaDokumentacja bibliotek przez MCP
COOLIFY_BASE_URLopcjaURL instancji Coolify
COOLIFY_ACCESS_TOKENopcjaToken Coolify
STITCH_API_KEYopcjaGoogle Stitch MCP server
GH_TOKEN_WORKopcjaToken do konta firmowego GitHub (SSO)
BACKUP_PINopcjaPIN do szyfrowania backupów

config/env-config.yaml — cała reszta

Tu definiujesz: tożsamość Git, listę pluginów, skills, serwery MCP. Podzielony na defaults (wspólne) i environments (per środowisko).

▶ Sekcja claude (język, uprawnienia)

Kontrolujesz język odpowiedzi Claude, tryb uprawnień i Remote Control:

claude:
  language: "Polish"
  remoteControl: false
  permissions:
    defaultMode: "bypassPermissions"

Tryb bypassPermissions oznacza, że Claude nie będzie pytać o zgodę na każde narzędzie.

▶ Dodaj plugin

Dopisz nazwę do listy plugins.marketplace:

plugins:
  marketplace:
    - superpowers
    - commit-commands
    - nowy-plugin        # ← dopisz tutaj

Po zmianie: przebuduj kontener lub uruchom setup-env.sh.

▶ Dodaj skill

Dopisz wpis do listy skills. Każdy wpis to jedno repo — instalator robi pojedyncze wywołanie npx skills add per repo:

skills:
  # pojedynczy skill
  - url: "https://github.com/autor/repo"
    name: nazwa-skill
  # bundle: wiele skilli z jednego repo, jedna instalacja
  - url: "https://github.com/anthropics/skills"
    names: [docx, pdf, xlsx]
  # wildcard: wszystkie skille z repo (dynamicznie)
  - url: "https://github.com/pbakaus/impeccable"

Dla wariantu wildcard rozwiązane nazwy skilli są śledzone w ~/.claude/skills/.sources.json, więc po usunięciu URL z listy sync automatycznie skasuje te skille.

▶ Dodaj serwer MCP

Dopisz w sekcji environments.<twój-env>.mcp_servers:

mcp_servers:
  - name: mój-serwer
    type: stdio
    command: npx
    args: ["-y", "@org/serwer@latest"]
    env:
      API_KEY: "${MÓJ_KLUCZ}"
Uwaga

Dla zdalnych serwerów używaj type: "http", nie "url" — "url" cicho nie działa.

▶ Konto firmowe GitHub

Jeśli masz oddzielne konto GitHub do pracy (SSO), środowisko automatycznie routuje tokeny:

git:
  work:
    email: "imie@firma.com"
    orgs: "TwojaOrg|InnaOrg"
    dirs:
      - "~/projects/Praca"

+ ustaw GH_TOKEN_WORK w .env.

Moduł 07

Ściągawka komend

Wszystko w jednym miejscu. Kliknij Copy i wklej do terminala.

Pierwsze uruchomienie

cp config/.env.example config/.env    # stwórz plik z tokenami
# edytuj config/.env — ustaw GH_TOKEN
claude mcp list                       # czy serwery MCP działają
claude plugin marketplace list        # czy pluginy się zainstalowały

Nowy projekt z loop

cd ~/projects/mój-projekt
loop init                             # zainicjalizuj loop
loop init --web                       # z webowymi skills
loop init --devops                    # z skills do Terraform

Codzienna praca

loop run -i "Dodaj autoryzację JWT"    # pomysł → plan → build
loop run --plan -i "Dodaj JWT"         # tylko faza planowania
loop run --build                       # tylko faza budowania (używa istniejącego planu)
loop design                           # interaktywny brainstorming
loop design -i "Pomysł na API"        # brainstorming z pomysłem
loop run --interactive                 # tryb ręczny
loop run --tmux                        # uruchom w sesji tmux
loop run --new -i "Następna rzecz"     # nowy cykl
loop kill                              # zbić procesy loop

Po zmianach w konfiguracji

./.devcontainer/setup-env.sh           # re-synchronizuj środowisko
loop update                           # odśwież symlinki loop
loop update --web --devops            # odśwież z presetów skills

Backupy

.devcontainer/backup.sh create         # zaszyfrowany backup
.devcontainer/backup.sh restore plik   # przywróć z backupu
.devcontainer/backup.sh list           # lista backupów
